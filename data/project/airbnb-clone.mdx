## 프론트엔드끼리의 좋은 협업을 위해

이번 프로젝트에서는, 백엔드 팀원은 없이 프론트엔드 두명이 팀을 이뤄 프로젝트를 진행하게 되었습니다. 프로젝트를 시작하기 앞서 팀원 앨런과 같이 **_우리의 좋은 협업을 위해 어떤 부분을 신경써야할까요?_** 라는 질문에 같이 고민해보기로 하였습니다. 이번 프로젝트는 사실, 코드스쿼드에서 진행했던 세번째 팀프로젝트였었는데요. 앨런과 이야기해보니 지금껏 이루어졌던 프로젝트의 협업과 관련해 앨런과 저 모두 아쉬움이 있었기 때문이었습니다. 그래서 저희는 다음과 같이 진행했습니다.

### 조금 더 좋은 소통을 이루기 위해

이를 위해, 저희는 저희 프로젝트만을 위한 노션페이지를 하나 만들어 협업규칙을 작성해 나가기로 하였습니다. 실제로 말로만 이루어지는 규칙보다 글로 적어가며 서로간에 이야기하는 방식에서 좋은 협업규칙을 만들 수 있었습니다. 저희의 협업규칙중 가장 중요한 규칙은 대화와 관련한 규칙이었습니다. 단 둘이서 이루어지는 프로젝트인 만큼 서로간에 방향성을 설정하고 토론할 일이 많았는데 토론을 할 때에 상대방의 의견을 무조건 수긍하지 않고 좋은점과 나쁜점을 말하기라는 규칙을 세웠습니다. 이로써 토론을 기반으로 한 협업을 진행하기로 하였고 이를 통해 프로젝트의 마지막까지 계속해서 토론을 해 좋은 프로젝트를 이루어낼 수 있었습니다.

두번째로 저희가 정했던 부분은 매일매일 조금씩이라도 회의하기였습니다. 앨런과 제가 모두 공감했던 부분이 팀프로젝트를 진행하면서 너무 바쁘다는 이유로, 또는 너무 피곤하다는 이유로 팀원간의 회의를 생략했던 일이 있었고 회의를 생략하자 서로간의 진행사항을 알기 힘들어 결과적으로 팀프로젝트가 아니고 개별적으로 분리된 프로젝트를 하는듯한 느낌이 들었다는 부분이었습니다. 이러한 문제를 해결하기 위해 저희는 매일 일정을 시작하기 전에 회의를 하고 회의록을 작성하기로 하였고 프로젝트 마무리까지 노션페이지에 회의록을 작성하며 진행하였고 조금 더 좋은 협업을 진행할 수 있었습니다.

### 코딩 컨벤션과 커밋 컨벤션 맞추기

서로간의 코드 스타일이 통일성을 갖출 수 있기를 원했습니다. 다른 사람과 협업을 할때에 사용하는 코드 스타일이 다를 경우 코드가 분리되어져 보이는 모습이 코드를 읽는 사람에게 좋지 않은 영향을 끼칠 수 있을 것이라 생각했기 때문입니다. 그래서, 코드 포맷터의 `prettier` 를 사용하고 코드의 포맷 형식을 맞추었습니다. 또 `ts lint` 를 `airbnb` 스타일로 진행하도록 하였습니다. 이를 통해 어느정도 코드 스타일을 맞출 수 있었습니다.

또한, 커밋컨밴션에 대해서도 신경 썼습니다. 우리가 기록을 남기고 있는 커밋 메세지들을 아무런 의미없이 작성한다고 생각하면 추후에 그 커밋 메세지를 보고 어떠한 내용이었는지 알기 힘들 것입니다. 이를 위해, 기능 추가나 기능의 수정, 스타일링 등을 구분해 커밋의 `prefix` 로 사용하고 상세한 커밋메세지를 적고자 규칙을 세우고 진행하였습니다.

### 페어프로그래밍

위와 같이 소통을 위해 노력하고 코딩 컨벤션을 맞추고 진행하고자 하였지만 때로 서로간의 코드를 이해하기 힘들거나 진행방식에 대한 조금더 구체적인 이야기가 필요한 순간들이 있었습니다. 프로젝트를 하던 당시에는, 코로나로 인해 대면으로 인한 진행이 어려웠기에 그러한 구체적인 소통이 조금 더 힘들었습니다. 이를 위해, 저희는 vscode의 라이브쉐어를 키고 페어프로그래밍을 자주 진행했습니다. 서로 코드를 치는 스타일을 보고 그러한 스타일에는 어떠한 장점이 있는지 배울 수 있었고 이를 통해 코딩컨벤션을 맞추고 서로간의 코드를 더욱 잘 이해할 수 있었습니다.

## 사용자의 좋은 경험을 위한 레이지로드와 스켈레톤 UI

이번 프로젝트에서 특히 신경을 쓰면서 진행했던 부분은 사용자의 좋은 경험성이었습니다. 지금까지 진행해왔던 프로젝트중 가장 많은 이미지와 데이터가 있었습니다. 그 데이터들을 특정 분기 조건마다 다르게 필터링해서 사용자에게 보여주어야 했습니다. 다량의 이미지들을 한꺼번에 불러온다면 그에대한 로딩시간이 많이 걸릴 수 밖에 없을 것이고 이에대한 처리를 어떻게 해야할 지 고민했습니다. 그래서, 저는 **_레이지로드와 스켈레톤 UI_**를 적용했습니다.

이미지를 lazyload하는 기능은 intersection observer를 사용해 사용자의 뷰포트에 이미지 미리 만들어둔 스켈레톤UI가 보여질때에만 이미지로드가 이루어지도록 했습니다. 이를 통해 사용자의 입장에서 필터링되는 시간을 단축시킬 수 있었습니다. 이번 프로젝트에서는 최대 500개의 이미지를 한꺼번에 불러와야하는 상황이었는데요. 사용자가 필요로하는 이미지만 불러오도록 해 조금 더 빠른 페이지 전환이 가능하게 되었고, 이를 통해 사용자의 편의성을 증진시킬 수 있었습니다. 또, 로딩스피너가 아닌 스켈레톤 UI를 적용해 이미지의 경우 로딩이 이루어지는 모습을 사용자가 직관적으로 볼 수 있도록 하였는데요. 이를 통해 로딩이 이루어지고 있다는 점을 사용자가 쉽게 확인할 수 있도록 했고 이러한 UI방식이 사용자가 기다릴 때 지루함을 줄일 수 있다는 것을 배울 수 있었습니다.

## contextAPI의 불편감을 느껴 적용했던 recoil

이번 프로젝트에서 처음에 적용했던 전역상태관리 라이브러리는 context API였습니다. 실제로 context API를 사용함으로 대부분의 기능을 문제없이 완료할 수 있었고 자체적으로 context API의 리렌더링 문제를 최소화하기 위해 context를 분리하는 등의 노력을 진행했었습니다. 그런데, context API를 계속해서 사용하던 중 두개의 context API가 동시에 필요로 하는 상태가 생길때마다 상위의 context를 만들어야 하는 상황에 직면했고 우리는 리렌더링 최적화와를 비롯 이러한 문제를 해결하기 위해 계속해서 다른 context를 만들어야 했습니다.

그래서, 제가 context API와 recoil를 비교해가며 어떠한 부분이 좋고 달라졌는지 확인해가며 일정 부분을 recoil로 리팩토링하는 부분을 전담했습니다. 이를 통해 recoil이라는 상태관리 라이브러리를 조금 더 공부할 수 있었고, `atom` 과 `selector` 등을 통해 젼역상태를 만들고 상태의 필터기능을 만들었습니다. 또한, context api의 고질적인 리렌더링 문제가 조금 완화되었음을 확인할 수 있었습니다.
